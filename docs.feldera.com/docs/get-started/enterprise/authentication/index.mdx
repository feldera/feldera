# Authentication

This guide describes how to configure authentication providers and tenant management for **Feldera Enterprise**.

Feldera supports enterprise-grade authentication through OIDC/OAuth2 providers, enabling secure multi-tenant deployments with flexible tenant assignment strategies.

## Supported Providers

Feldera supports the following authentication providers:

- **AWS Cognito** - Enterprise identity management with Cognito User Pools
- **Okta** - Enterprise SSO integration for B2B deployments
- **Google Identity** - Google OAuth2 authentication
- **None** - No authentication (development/testing only)

## Tenant Assignment Models

Feldera provides flexible tenant assignment strategies to support different deployment patterns:

### Individual Tenancy (Default)
Each authenticated user gets their own private tenant based on their `sub` claim. Ideal for development and small team deployments.

### Organization-wide Tenancy
Users from the same organization share a tenant, enabling collaboration and resource sharing. Configured through explicit tenant claims or automatic issuer domain extraction.

### User Group Tenancy
Multiple teams can use the same Feldera instance with complete tenant isolation. Each team's users should be assigned to a corresponding user group with the proper configuration of a dynamic tenant claim.

## Tenant Assignment Strategies

Configure tenant assignment behavior using a combination of the following strategies:

### Individual Tenancy

Controls whether individual users can create their own tenants based on their OIDC user ID (`sub` claim). No additional IDP configuration required.

**Feldera Configuration:**
```bash
pipeline-manager ...\

# Enable individual tenancy (default) - a tenant is created automatically for every new user
--individual-tenant=true

# Disable individual tenancy (enterprise mode)
--individual-tenant=false
```

### Organization Tenancy (one tenant per org)

Lets all users from your Okta org share one tenant based on the organization domain.

The tenant name is extracted from OIDC issuer domain:
- `https://acme-corp.okta.com/oauth2/default` → `acme-corp` tenant

**Feldera Configuration:**
```bash
pipeline-manager ...\

# Enable organization-based tenancy
  --issuer-tenant=true

# Disable organization-based tenancy (default)
  --issuer-tenant=false
```

Example tenant names:
- `https://acme-corp.okta.com/oauth2/default` → `acme-corp` tenant
- `https://company.auth.us-west-2.amazoncognito.com` → `company` tenant

## Tenant Resolution Priority

Feldera resolves tenant assignment using the following priority order:

1. **`tenant` claim** - Explicit tenant assignment via OIDC provider
2. **Issuer domain extraction** (when `--issuer-tenant=true`)
3. **User `sub` claim** (when `--individual-tenant=true`)

If no valid tenant is found and `--individual-tenant=false`, the user will be denied authorization.

## Common Deployment Patterns

### Development Environment
```bash
# Default settings - each user gets individual tenant
pipeline-manager --auth-provider=none
```

### Single Organization
```bash
# All users from organization share one tenant
pipeline-manager \
  --auth-provider=okta \
  --issuer-tenant=true \
  --individual-tenant=false
```

### Multi-Customer B2B
```bash
# Each customer organization gets separate tenant
pipeline-manager \
  --auth-provider=okta \
  --issuer-tenant=true \
  --individual-tenant=false
```

### Custom Tenant Claims
```bash
# Use explicit tenant claims from OIDC provider
pipeline-manager \
  --auth-provider=okta \
  --individual-tenant=false
```
